## 方式
- URL本体は振動で送らない。**Supabase で各URLに数値id（整数）を割り当て**、**idだけ**を振動で送る。
- 受信側は id を復元 → DBで逆引き → URLを開く。
- 振動は帯域が細くURL全体は送れないため、番号方式に確定。
- **手動演奏が体験の主役**: 画面の楽譜を見ながら人間が「短/長」ボタンで演奏する。自動演奏も併設（デモ保険・受信チューニング基準）。

## ビット表現
- 短い振動（ブッ）= `0` / 長い振動（ブーッ）= `1`
- ビット順: **MSB first**（最上位ビットから送る）
- **id ビット長: 8ビット（固定長、0〜255）**
- **終端マーカー: なし**（固定長のため、モードマーカー+8ビット受けたら終了）

## フレーム構成
```
[プリアンブル] → [モードマーカー 1bit] → [ペイロード]
```
- **モードマーカー**: `0` = idモード（本線）/ `1` = X1直接符号化モード（短URL専用）
- idモードのペイロード = id 8ビット（MSB first）→ **1送信 = プリアンブル + 9打**
- X1モードのペイロード = 下記 `## X1 フレーム（直接符号化モード / v1.1）` 参照

## X1 フレーム（直接符号化モード / v1.1）
短URLを id 逆引きなしで振動だけで送る。**marker=0（idモード）は一切変更しない（回帰なし）。**

```
[プリアンブル] [marker=1] [scheme 1bit] [length 6bit] [chars: length×6bit] [checksum 8bit]
```
各フィールド **MSB first**、ビット表現は idモードと共通（短=0 / 長=1）。

- **marker** … `1`（長振動）= URL直接モード。
- **scheme** … `0`=https / `1`=http。受信側は `scheme://` + 復元文字列 を開く。
- **length** … chars の文字数。6bit のため **最大 63**（X1は短URL専用）。
- **chars** … 1文字=6bit。下記テーブルのインデックス。
- **checksum** … chars のインデックスバイト列に対する **CRC-8**。検出のみ・訂正はしない。

### 6bit 文字テーブル（送受信で完全一致させる）
| idx | 文字 |
|---|---|
| 0–25 | `a`–`z` |
| 26–35 | `0`–`9` |
| 36–58 | `.` `-` `_` `~` `:` `/` `?` `#` `[` `]` `@` `!` `$` `&` `'` `(` `)` `*` `+` `,` `;` `=` `%`（この順で23個） |
| 59–63 | 予約（未使用） |

対応外文字（大文字・レア記号）はデモURLで避ける（小文字前提）。シフト符号は後回し。

### checksum（CRC-8）の定義【確定】
- 多項式 **poly = 0x07** / 初期値 **init = 0x00** / **無反転（reflect なし）** / **xorout なし**。
- 入力は **chars 各文字の6bitインデックス値（0〜63）を1バイトずつ** MSB first で投入する（index=byte で一意化）。
- ※簡易版として **sum mod 256**（インデックス総和の下位8bit）も許容するが、本線は CRC-8。

### 失敗時の挙動（方針b・確定）
- checksum OK → `scheme://` ＋ 復元文字列 を開く。
- checksum NG → 「ミスったので運命のサイトへ🎲」表示 ＋ 受信側の FUN_SITES からランダムに1つ開く。

### 演奏方法
- 自動演奏X1 … 機械が完璧に弾く＝必ず成功（確実な見せ方）。
- 人間演奏X1 … 音ゲーで長い曲として叩く＝たまに失敗→運命サイトが発動（ネタの見せ場）。
- F11 は注入式のため、可変長の X1 パルス列もそのまま流せる。

## タイミング定数【確定】
| 要素 | 値 |
|---|---|
| 短い振動 (0) | **150ms ON** |
| 長い振動 (1) | **450ms ON** |
| ビット間 gap | **150ms OFF** |
| プリアンブル | **[700ms ON, 200ms OFF] × 2** |

- プリアンブルONを700msにした理由: 長(450ms)との混同を避けるため。
- 手動演奏時も、ボタン1打で出る振動は上記の固定長（短ボタン=150ms、長ボタン=450ms）。
  **人間がブレるのは「打つ間隔」だけ**であり、パルス長自体は常に機械精度で出る。

## 復調規則（受信側）
1. プリアンブル（700ms級のON×2）を検出して信号開始と単位時間を得る。
2. 以降のONパルス長を測り、短/長を判定して 0/1 に変換（MSB first で再構成）。
   - 自動演奏: 絶対値（150/450ms±許容）で判定可。
   - 手動演奏: **比率ベース**で判定（パルス長は固定なので主にgapの揺れを許容する。
     ONパルス長 300ms を境に 短/長 を分ければ両モード共通でよい）。
3. モードマーカー1bit → ペイロード8bit を受けたら1メッセージ完了。
4. ビット間gapの揺れ（手動演奏）は判定に使わない。**ONパルス長のみで0/1を決める。**

## 人間／機械 両対応
- 自動演奏も手動演奏も「スマホの振動」として全く同じ媒体で発される。受信パイプラインは1本。
- 手動演奏では打つ間隔が大きく揺れるため、受信側はgapに依存しない設計とする（上記復調規則）。

## 受信環境（前提）
- ラップトップにスマホを直置き（接触・近距離）を基本とする。
- 第一センサー: PCマイク（44.1kHz）。補助: 外付けIMU / ピエゾ。
- 検出律速はモーター立ち上がり（最短150ms）。100Hz以上のサンプリングで検出は十分。

## 所要時間の目安（idモード）
- プリアンブル ≈ 1.8s ＋ 9打 ×（平均450ms）≈ 4s → **1送信 約6秒**（手動はテンポ次第）。
