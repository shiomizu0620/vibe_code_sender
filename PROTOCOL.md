> 各実装に定数を二重に書かない（ここからコピーし、変更時はここを起点にする）。

## 方式
- URL本体は振動で送らない。**Supabase で各URLに数値id（整数）を割り当て**、**idだけ**を振動で送る。
- 受信側は id を復元 → DBで逆引き → URLを開く。
- 振動は帯域が細くURL全体は送れないため、番号方式に確定。
- **手動演奏が体験の主役**: 画面の楽譜を見ながら人間が「短/長」ボタンで演奏する。自動演奏も併設（デモ保険・受信チューニング基準）。

## ビット表現
- 短い振動（ブッ）= `0` / 長い振動（ブーッ）= `1`
- ビット順: **MSB first**（最上位ビットから送る）
- **id ビット長: 8ビット（固定長、0〜255）**
- **終端マーカー: なし**（固定長のため、モードマーカー+8ビット受けたら終了）

## フレーム構成
```
[プリアンブル] → [モードマーカー 1bit] → [ペイロード]
```
- **モードマーカー**: `0` = idモード（本線）/ `1` = 直接符号化モード（stretch・自動演奏限定）
- idモードのペイロード = id 8ビット（MSB first）→ **1送信 = プリアンブル + 9打**
- 直接符号化モードのペイロード = 別途定義（stretch着手時にこのファイルへ追記）

## タイミング定数【確定】
| 要素 | 値 |
|---|---|
| 短い振動 (0) | **150ms ON** |
| 長い振動 (1) | **450ms ON** |
| ビット間 gap | **150ms OFF** |
| プリアンブル | **[700ms ON, 200ms OFF] × 2** |

- プリアンブルONを700msにした理由: 長(450ms)との混同を避けるため。
- 手動演奏時も、ボタン1打で出る振動は上記の固定長（短ボタン=150ms、長ボタン=450ms）。
  **人間がブレるのは「打つ間隔」だけ**であり、パルス長自体は常に機械精度で出る。

## 復調規則（受信側）
1. プリアンブル（700ms級のON×2）を検出して信号開始と単位時間を得る。
2. 以降のONパルス長を測り、短/長を判定して 0/1 に変換（MSB first で再構成）。
   - 自動演奏: 絶対値（150/450ms±許容）で判定可。
   - 手動演奏: **比率ベース**で判定（パルス長は固定なので主にgapの揺れを許容する。
     ONパルス長 300ms を境に 短/長 を分ければ両モード共通でよい）。
3. モードマーカー1bit → ペイロード8bit を受けたら1メッセージ完了。
4. ビット間gapの揺れ（手動演奏）は判定に使わない。**ONパルス長のみで0/1を決める。**

## 人間／機械 両対応
- 自動演奏も手動演奏も「スマホの振動」として全く同じ媒体で発される。受信パイプラインは1本。
- 手動演奏では打つ間隔が大きく揺れるため、受信側はgapに依存しない設計とする（上記復調規則）。

## 受信環境（前提）
- ラップトップにスマホを直置き（接触・近距離）を基本とする。
- 第一センサー: PCマイク（44.1kHz）。補助: 外付けIMU / ピエゾ。
- 検出律速はモーター立ち上がり（最短150ms）。100Hz以上のサンプリングで検出は十分。

## 所要時間の目安（idモード）
- プリアンブル ≈ 1.8s ＋ 9打 ×（平均450ms）≈ 4s → **1送信 約6秒**（手動はテンポ次第）。
