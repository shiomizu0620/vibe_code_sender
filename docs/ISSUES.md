## [Vibes] Flutter 送信側

### F1: 振動の基盤実装（constants / vibrator_service）
**担当: A（低レイヤ）** / 触ってよい: `lib/constants.dart`, `lib/vibrator_service.dart` のみ
- PROTOCOL.md v1.0 の定数（150/450/150ms, プリアンブル[700,200]×2）を `constants.dart` に定義
- `vibrator_service.dart`: `hasVibrator` チェック＋生パターン配列の再生 `play(List<int>)`
- 受け入れ: 任意のパターン配列を渡すと振動が再生される
- 依存: なし（即着手可）

### F2: pattern_builder 実装
**担当: A** / 触ってよい: `lib/pattern_builder.dart` のみ（constantsは参照のみ）
- `enum Pulse { short, long }`、Pulse列 → 振動パターン配列（[待ち,振動,...]）
- 先頭にプリアンブル[700,200]×2 を付与する関数
- 受け入れ: `[short,long,short]` から正しいms配列が決定的に生成される（ユニットテスト付き）
- 依存: F1

### F3: encoder 実装（id → Pulse列）
**担当: B（UI/ロジック）** / 触ってよい: `lib/encoder.dart` のみ
- `List<Pulse> encode(int id)`: id(0..255) → モードマーカー(0=short) + 8bit MSB first
- 範囲外idはエラーを投げる（握りつぶさない）
- 受け入れ: `encode(42)` が決定的に9個のPulseを返す（ユニットテスト付き）
- 依存: F2（Pulse定義）。なおF2完了前でもenum定義合意のうえ並行可

### F4: 楽譜表示UI ★体験のコア
**担当: B** / 触ってよい: `lib/score_view.dart`(新規), `lib/main.dart`
- id→Pulse列を「●(短)/━(長)」等の記号列で表示
- 演奏位置カーソル: 何打目まで打ったかをハイライト
- 受け入れ: idを与えると楽譜が表示され、進捗に応じてカーソルが進む
- 依存: F3

### F5: 手動演奏モード（短/長ボタン）★体験のコア
**担当: B** / 触ってよい: `lib/main.dart`, `lib/score_view.dart`
- 「短」ボタン → 150ms振動1発、「長」ボタン → 450ms振動1発（vibrator_service経由）
- 打つたびに楽譜カーソルが進む。プリアンブルは演奏開始ボタンで自動送出（人間に700msは打てないため）
- 受け入れ: 楽譜を見ながらボタンで最後まで演奏でき、各打で正しい長さの振動が出る
- 依存: F1, F4

### F6: 自動演奏モード
**担当: A** / 触ってよい: `lib/main.dart` の自動演奏ボタン部のみ（Bと衝突注意・要一言）
- id → encode → pattern_builder → 一括再生（プリアンブル込み・機械精度）
- 受け入れ: id入力→ボタン1つでフル信号が正確に再生される
- 依存: F2, F3

### F7: id入力UI（テスト用）
**担当: B** / 触ってよい: `lib/main.dart`
- 数値入力欄(0..255) → 楽譜表示＋手動/自動演奏に渡す
- 受け入れ: 任意のidで楽譜生成・演奏ができる
- 依存: F4

### F8: 実機での振動確認
**担当: A** / コード変更なし
- Android実機を確保し、短/長/プリアンブルが体感・録音で区別できることを確認
- 受け入れ: 実機録音で3種のパルス長が判別できる（受信側チューニングの材料として共有）
- 依存: F1, F2

### F9: Supabase連携（URL登録→id発行）
**担当: A or B（着手時に決定）** / 触ってよい: `lib/supabase_service.dart`(新規), `pubspec.yaml`(要宣言), `lib/main.dart`(登録画面)
- `urls(id, url, created_at)` への登録、id発行、一覧取得
- URL一覧から選ぶ→楽譜表示、の動線
- 受け入れ: URLを登録するとidが発行され、一覧から選んで演奏できる
- 依存: F4, F7。共通S1（Supabaseプロジェクト作成）

## [receiver] Python 受信側（担当: リーダー）

### P1: 環境構築
- Python + `sounddevice` + `numpy`/`scipy`。マイクからリアルタイムで波形が取れることを確認
- 受け入れ: 録音した波形をプロットできる

### P2: 包絡線抽出→ON/OFF二値化
- バンドパス（モーター周波数帯はF8の録音から決定）→ 整流/平滑 → 閾値でON区間検出
- **手叩き・任意のスマホ振動でテスト可（Flutter完成を待たない）**
- 受け入れ: 振動のON区間の開始/終了タイムスタンプ列が出力される

### P3: プリアンブル検出
- 700ms級ON×2 のパターンを検出して信号開始を確定
- 受け入れ: プリアンブル後の最初のデータパルスから計測が始まる

### P4: 復調（パルス長→0/1→id復元）
- ONパルス長 300ms境界で短/長判定 → モードマーカー1bit＋8bit MSB first → id復元
- **検出と復号を分離**: 手打ちのタイムスタンプ配列を入力にしたユニットテストを書く
- gapの揺れは判定に使わない（PROTOCOL.md 復調規則）
- 受け入れ: タイムスタンプ配列→idのテストが通る／実音声からもidが出る

### P5: Supabase逆引き→URLオープン
- `supabase-py` で id→url を引き、`webbrowser.open(url)`
- 受け入れ: id=42 を与えると登録済みURLがブラウザで開く
- 依存: 共通S1

### P6: ターミナル演出
- 受信中: ビットが1つずつ並んでいく表示（●━...）→ id確定 → URLをタイプ風に1文字ずつ表示 → オープン
- `rich` 推奨
- 受け入れ: 受信〜オープンまでが演出付きで流れる

### P7: デバッグ可視化
- 波形・包絡線・検出パルス・判定結果をプロット表示（matplotlib等）
- 結合チューニング(J2)で必須になるため、P2〜P4と並行で育てる
- 受け入れ: 1回の受信について何がどう判定されたか目で追える
